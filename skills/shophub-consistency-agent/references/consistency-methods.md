# 一致性检查方法选择

本参考用于选择能证明规格的最小方法。工具是手段，规格 ID 和可复现证据是验收核心。

## REST 契约

- 以 README 和设计附录生成设计侧 OpenAPI；不得以 Controller 生成结果替代事实源。
- 静态比较适合发现路径、方法、字段和状态码漂移。
- 真实 HTTP 黑盒负责验证运行时认证、序列化、异常映射和副作用。
- Schemathesis 可用时用于补充边界和非法输入；认证、前置数据或状态关系无法自动推导时仍需手写 JUnit。

## 状态模型

- 状态和转换较少时使用 JUnit 参数化转换矩阵，失败定位最直接。
- 跨订单、支付、退款和物流产生组合路径时，可使用 GraphWalker 等模型工具生成路径。
- 模型必须来自设计，且为每个转换定义可观察断言；只遍历接口不检查结果无效。

## 属性测试

- 对金额、库存、优惠、积分和退款编写不变量：非负、上界、舍入、幂等、守恒和顺序。
- jqwik 可直接接入 JUnit Platform；依赖不可用时使用 JUnit 参数化边界集合。
- 不要在测试中复制生产算法作为预言。优先使用设计公式、独立简单计算或关系不变量。
- 记录随机种子或失败样本，确保反例可复现。

## 架构约束

- 使用 ArchUnit 检查 Controller 不访问 Repository、模块不访问其他模块 Repository、公共模块不反向依赖业务模块、跨模块不暴露 JPA Entity，以及设计禁止的依赖方向。
- 当前 Maven 模块已表达粗粒度边界，优先使用 ArchUnit；除非项目已经采用 Spring Modulith，不为单次检查迁移包结构。
- 无法引入依赖时可用 `jdeps`、源码扫描或 Maven 依赖图降级，但必须报告静态检查的盲区。

## 变异测试

- PIT 用于验证关键测试能否识别条件反转、算术变化和返回值错误，不直接判断设计正确性。
- 只针对状态判断、金额计算、权限和库存等关键包，避免全仓库样板代码稀释信号。
- 存活变异先判断是否等价，再补强测试；不得单纯追求百分比。

## 不优先采用的方法

- Pact 和 Spring Cloud Contract 适合独立消费者与提供者协作。ShopHub 是同进程模块化单体，现阶段不能替代业务黑盒。
- REST 模糊测试擅长发现 5xx 和健壮性问题，但不能单独判断金额或业务状态是否正确，只作为契约测试补充。

## 交付证据

每种方法都必须落到以下至少一项：失败测试、违规报告、最小反例、契约差异或变异存活记录。没有可复现产物的人工判断只能列为风险，不能记为已验证。
