# ShopHub 一致性 Skill 优化与 A/B 验证报告

日期：2026-07-13

## 1. 结论

优化后的候选流程显著提升了规格资产的广度、可追踪性和架构问题发现能力，但首次 B 组执行也暴露出一个关键缺陷：**映射数量和测试执行次数不能代表有效语义覆盖**。B 组生成了 82 次黑盒执行和 87 条追踪映射，但其中 81 条接口用例主要只排除 405/5xx，6 条高风险设计规格中只有注册/未激活登录两项得到真实业务断言。

因此，不能据此宣称 B 在缺陷检出深度上全面优于 A。受控变异实验中两组均只杀死 1/3 个变异，能力互补：A 发现管理接口路径漂移，B 发现注册状态回退，两组均漏掉创建订单成功状态码回退。

本轮已经把这一发现反馈到最终 Skill：强制保留并审核全部 659 条设计候选，禁止把路由探测计为契约验收，要求逐接口精确成功状态码、路径和关键业务证据，并验证证据确实存在于被引用的测试方法体。最终严格校验器会以 914 项错误拒绝首次 B 组的浅层产物。

## 4. 缺陷发现差异

两组共同发现并修复：

- 注册状态和激活令牌错误；
- 未激活用户登录未被拒绝；
- 支付初始状态错误；
- 八折券计算方向错误；
- 订单应付金额遗漏运费。

A 组额外深入发现并修复：

- 优惠券状态、模板状态和有效期校验；
- 支付金额必须与订单应付金额完全一致；
- 退款等待仓库验收状态及退款公式；
- 部分开票和 6% 税率；
- 未公开且危险的重置/引导管理员接口；
- 更多异常状态码映射与对应 UT。

B 组额外发现并修复：

- 支付模块绕过模块边界、直接以 `JdbcTemplate` 查询订单表，改为通过 `OrderQueryService`；
- 创建订单 UT 的过时 200 断言与冻结契约 201 不一致；
- Mockito 在当前 JDK 环境中的 mock maker 兼容问题。

这表明规格先行流程对结构性漂移有实际价值，但自动生成的契约用例如果没有强预言，只会扩大“被调用的接口数”，不会同步扩大业务缺陷发现深度。

## 5. 受控变异实验

只运行各组自己生成的 `test-case-new/`，不运行原始公开黑盒，以测量新增验收的独立识别能力。

| 变异 | A | B |
| --- | --- | --- |
| 注册状态 `PENDING_ACTIVATION → ACTIVE` | 存活，4/4 | 杀死，82 中 1 失败 |
| 配置接口 `/configs/{key} → /configs-hidden/{key}` | 杀死，4 中 1 失败 | 存活，82/82 |
| 创建订单 `201 → 200` | 存活，4/4 | 存活，82/82 |

因此两组均为 33.3%，但覆盖的风险不同。B 的 81 条参数化接口探测对路径漂移不敏感，是最终 Skill 强制检查精确状态码和路径证据的直接原因。

## 6. 行业方法调研及本项目适配

| 方法 | 官方能力依据 | ShopHub 适配结论 |
| --- | --- | --- |
| 设计侧 OpenAPI + HTTP 契约 | [Schemathesis 官方文档](https://schemathesis.readthedocs.io/en/stable/)说明可从 OpenAPI 生成属性测试和边界输入 | 适合补充输入空间和 5xx/结构问题；认证、前置数据、金额和状态语义仍需手写 JUnit 预言 |
| ArchUnit | [ArchUnit 官方指南](https://www.archunit.org/userguide/html/000_Index.html)支持依赖、分层、环和包规则检查 | 非常适合 ShopHub 多 Maven 模块的 Repository 越界、Controller 分层和模块依赖规则；B 已用人工静态检查发现一处真实跨模块直查 |
| 属性测试 | [jqwik 官方指南](https://jqwik.net/docs/current/user-guide)支持 JUnit Platform、自动生成参数和失败样本缩减 | 适合优惠、退款、积分、库存的非负、上界、守恒、舍入和幂等规则；预言必须来自设计公式，不能复制生产算法 |
| 变异测试 | [PIT Maven 官方指南](https://pitest.org/quickstart/maven/)支持限定目标类和目标测试 | 适合关键状态判断、金额公式和权限分支；本轮受控实验已经证明“测试数量多”并不等于能杀死关键错误，应定向使用而非追求全仓库百分比 |

不建议把消费者契约工具作为主方案：ShopHub 是同进程模块化单体，当前主要风险是设计规则、业务状态和模块边界漂移，而不是独立团队维护的远程消费者/提供者协议。

## 7. 最终 Skill 的流程改进

最终版本采用以下闭环：

1. 冻结 `README.md`、`design-docs/`、`test-cases/` 的 Git 跟踪文件摘要。
2. 先从 README 生成 81 个 REST 契约，再从全部设计文档抽取 659 条候选。
3. 逐项审核候选为 `required` 或有具体理由的 `informational`，不得删项。
4. 只读探索原始黑盒，并以覆盖缺口生成独立 `test-case-new/`。
5. 对每个 API 要求真实 HTTP、合法前置条件、精确成功状态码、关键字段或副作用；405/5xx 探测只能标记为 `routing`。
6. 对每条映射校验规格专属证据确实位于对应测试方法体。
7. 根据规则类型选用状态矩阵、属性测试、ArchUnit 和定向 PIT。
8. 按“失败证据 → 根因修复 → 对应 UT → 相关黑盒 → 全量回归”循环，直至严格追踪、UT、原黑盒和新增黑盒全部通过。

Skill 内已删除对比测试内容；A/B 验证仅保留在本报告和 `/tmp` 实验工作区。

## 8. 限制与后续验收建议

- 本次是每组一次完整独立运行，不是多次随机重复实验，不能给出统计显著性结论。
- B 组运行的是首次优化候选；严格校验规则是在 A/B 暴露伪覆盖后形成的最终版本。最终版本已经通过静态和样例校验，并成功拒绝 B 组产物，但尚未进行第三次完整端到端修复运行。
- 两组最终源码分支都通过各自 UT 和公开黑盒，但修复集合不同；不能把任一临时分支直接视为最终合并方案。
- 若继续执行项目最终修复，应以最终严格 Skill 重新生成完整 `spec-inventory.json` 和语义黑盒，并重点复核支付异步后置动作、回调签名、退款、发票、优惠券有效期及危险管理接口。

综合判定：**流程质量有明确提升，缺陷检出深度尚无整体提升证据。** 最终 Skill 已针对实验暴露的问题完成加固，下一轮完整执行应以严格追踪校验通过作为进入源码修复和最终交付的前置条件。
